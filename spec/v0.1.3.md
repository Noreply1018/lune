# v0.1.3 Plan: Integrated Lune Image

Status: accepted direction, pending implementation cleanup

## Goal

Make Lune feel like one product and one Docker image for end users.

The desired default experience is:

- Users pull and run a single Lune image.
- Lune backend, admin UI, and CPA capability are available together.
- Users do not need to understand or operate a second CPA image.
- Configuration and persistence are simpler than the current two-service Compose setup.

## Current Problem

Lune currently ships as one application image plus a Compose-managed CPA service image.

This is technically clean at the container boundary, but it creates product friction:

- Docker Desktop users cannot get the full Lune experience by pulling and running only `noreply1018/lune`.
- CPA configuration is split across Lune environment variables, `cpa-config.yaml`, shared volumes, and Compose service wiring.
- New users have to understand that Lune is not complete without CPA.
- The distribution story depends on another image being available and compatible.

## Key Questions

### CPA Update Cadence

If Lune embeds CPA into its image, Lune no longer automatically receives CPA updates just because `eceasy/cli-proxy-api:latest` changes.

The release owner must decide how CPA updates flow into Lune:

- Pin CPA to a known version per Lune release for reproducibility.
- Regularly bump the embedded CPA version in Lune releases.
- Optionally expose an advanced external CPA mode for users who need latest CPA independently.

### Product Boundary

The accepted v0.1.3 product boundary is:

- Lune image includes the CPA runtime.
- Lune owns the default CPA configuration.
- CPA is an implementation detail for the default user path.
- External CPA mode is not part of the v0.1.3 default scope.

### Configuration Model

If integrated, the default config should collapse into Lune-owned settings:

- One data directory.
- One auth directory under that data directory or a clearly named subdirectory.
- One set of environment variables.
- No required `cpa-config.yaml` for the default path.
- No shared Docker volume between services, because there is only one container.

### Runtime Model

There are several possible implementation routes:

- Start both Lune and CPA as two processes inside one container.
- Make Lune supervise CPA as a child process.
- Vendor or reimplement the CPA server functionality directly in Go.
- Keep CPA external, but provide a separate all-in-one image.

Each route has different maintenance cost and failure behavior.

## Accepted Direction

v0.1.3 should make the main Lune image integrated by default:

- `noreply1018/lune` is all-in-one by default.
- It starts Lune and embedded CPA together.
- It auto-generates CPA config from Lune environment variables.
- It stores data in `/app/data`.
- It stores the SQLite database at `/app/data/lune.db`.
- It stores CPA auth files under `/app/data/cpa-auth`.
- It does not require `cpa-config.yaml` for the default path.
- It pins the embedded CPA version per Lune release.

## Decisions Made

- `noreply1018/lune` should become the all-in-one default image.
- CPA should be embedded as part of the Lune image.
- CPA should be pinned per Lune release, not floated implicitly for end users.
- v0.1.3 should not auto-replace the embedded CPA binary at runtime.
- v0.1.3 should check for newer CPA releases and surface an update hint instead.
- `cpa-config.yaml` should be removed from the public quick start.
- The default persistence model should be one mounted data directory.
- Docker Desktop first-run product wording should require no environment variables.
- `CPA_API_KEY` and `LUNE_CPA_MANAGEMENT_KEY` are internal embedded-runtime credentials in the default path, not user-facing setup inputs.
- `LUNE_ADMIN_TOKEN` is optional for local personal use and recommended when exposing Lune through remote access, reverse proxy, VPN, or untrusted networks.
- Pool should become the only external client access boundary.
- Global access tokens should be removed from the product model.
- Each Pool should own exactly one default-enabled access token.
- Client setup should be generated from a Pool context, not from a global token.

## Remaining Decisions

- Should CPA be pinned by digest or immutable version tag?
- What migration behavior should existing users get from `http://cpa:8317` to embedded CPA?
- How should logs distinguish Lune logs from CPA logs in one container?

## Deferred

Advanced external CPA mode is deferred to the next tag draft.

Stricter remote access authentication is deferred to a later tag. v0.1.3 keeps the current local-first behavior, but documentation must warn users not to expose the admin port directly to untrusted networks.

The spike implementation proves the integrated image can work, but it needs cleanup before becoming v0.1.3.

Required adjustments:

- Pin the CPA image source. Do not build from `eceasy/cli-proxy-api:latest` for a release.
- Use one persistence mount by default: `/app/data`.
- Move CPA auth files to `/app/data/cpa-auth`.
- Remove the extra `cpa-auth` Docker volume from the default Compose files.
- Do not publish CPA port `8317` by default unless there is a user-facing reason.
- Keep `127.0.0.1:8317` as an internal container URL for Lune-to-CPA communication.
- Update Docker Desktop instructions around a single port mapping: host `7788` to container `7788`.
- Rework UI copy that still says users must configure CPA in Settings.
- Redesign the Settings CPA area as an embedded runtime status section that matches Lune's existing visual style.
- Keep CPA wording in Add Account and account cards; only change Add Account's missing-service error state.
- Ensure first-run auto-configuration always creates a usable CPA service row.
- Ensure migration from the old Compose URL `http://cpa:8317` updates existing default CPA service rows.
- Add tests for default CPA auto-configuration and old URL migration.
- Prefix or otherwise distinguish CPA logs from Lune logs in the single container.
- Decide whether `cpa-config.yaml` should be deleted entirely or kept only as legacy reference.
- Fix gateway early-error observability for oversized and malformed request bodies.
- Revisit the fixed 10MB gateway request body limit; `/v1/responses` and image-heavy payloads can exceed it in normal use.

Nice-to-have adjustments:

- Show the embedded CPA version in startup logs, Settings, or release notes.
- Show whether a newer CPA release is available, but direct users to upgrade the Lune image for the default path.
- Add a health check that verifies both Lune and embedded CPA are reachable.
- Add a release checklist item for bumping embedded CPA.

## Embedded CPA Version Policy

v0.1.3 should treat the embedded CPA binary as part of the Lune release artifact.

Default policy:

- The embedded CPA version is pinned by the Lune release.
- Lune should not download and execute a newer CPA binary automatically.
- Users get a newer embedded CPA by upgrading the Lune image.
- Lune should make the current embedded CPA version visible.
- Lune may periodically check the latest upstream CPA release and show a non-blocking update notice.

Rationale:

- The same Lune image should behave reproducibly over time.
- Support and debugging should be able to map behavior to a Lune version.
- Runtime binary replacement adds security and rollback complexity.
- Automatic CPA updates could change model routing behavior without a Lune release.

Suggested v0.1.3 UI/API surface:

- Settings CPA runtime section shows `Current CPA Version`.
- Settings CPA runtime section shows `Latest CPA Version` if the check succeeds.
- If a newer CPA exists, show concise copy such as `A newer embedded CPA is available in a newer Lune image.`
- Do not expose a default "Update CPA" action in v0.1.3.

Suggested implementation details:

- Detect current CPA version from the embedded CPA process or startup probe output.
- Check upstream release metadata on a conservative interval.
- Cache the latest-version result so Settings does not depend on GitHub/Docker Hub availability.
- If the check fails, keep the runtime section usable and show the current version only.
- Include the pinned CPA version in release notes and Docker image labels where practical.

## Gateway Large Payload Issue

Current behavior:

- Gateway requests are fully read into memory before routing.
- The body is capped at 10MB.
- Requests over the cap return `413 request body exceeds 10MB limit`.
- This can happen during normal `/v1/responses` usage when image or file payloads are embedded in the request.
- These early failures are not inserted into `request_logs`, because logging currently happens after body parsing or route resolution.
- The HTTP access log does not include status code, so stdout logs are not enough to diagnose the 413.

Required v0.1.3 fixes:

- Make the maximum gateway request body size configurable.
- Pick a larger default that works for common image-heavy requests.
- Log early gateway failures such as oversized body, malformed body, and missing model into `request_logs`.
- Include HTTP status code in access logs.

Accepted design:

- Keep a configurable maximum gateway request body size.
- Buffer small requests in memory.
- Spill larger requests to a temporary replay file.
- Re-open the replay file for each retry attempt.
- Delete the temporary file when the request finishes.
- Store temporary replay files under the Lune data directory, for example `/app/data/tmp`.

Suggested defaults:

- `gateway_max_body_mb`: 100
- `gateway_memory_body_mb`: 8
- `gateway_tmp_dir`: `/app/data/tmp`

Settings UI:

- Add payload controls to the existing Settings Runtime section.
- Place them near `Request Timeout` and `Max Retry Attempts`, because they define gateway execution behavior.
- Add `Max Request Body` as a numeric field with `MB` suffix.
- Add `Memory Body Threshold` as a numeric field with `MB` suffix.
- Keep the temporary directory out of the default UI unless there is a concrete user need; it should be documented as an environment/default path.
- Helper copy should explain that requests above the memory threshold are buffered on disk for retry instead of kept fully in memory.
- Preserve the visual balance of the Settings first section. Runtime currently sits beside the CPA section; adding two rows must not make the left column feel overloaded or leave the right column visually empty.
- If needed, redesign the two-column area as balanced grouped panels rather than simply appending rows.
- Keep compact Settings typography and spacing; do not introduce a large explanatory card.

## Frontend Surface Decisions

Settings should change from "configure CPA Service" to "observe embedded CPA runtime".

Add Account should still call the source "CPA". The only required copy change is the missing-service/error state:

- Current: "请先在 Settings 配置 CPA Service。"
- Target: "内置 CPA 服务未就绪，请稍后重试或检查容器日志。"

Account cards and account detail surfaces should keep the current CPA semantics for v0.1.3.

## Additional v0.1.3 Scope

These are adjacent to the integrated-image and large-payload work and are in scope for v0.1.3.

### Gateway Observability

- Capture response status in the HTTP access logger.
- Log gateway early failures consistently: invalid token can remain access-log only, but authenticated malformed/oversized gateway requests should appear in Activity.
- Include request size and whether the body used memory or disk replay in debug logs or Activity details if the schema/UI can absorb it cleanly.

### Model Parsing For Large Requests

The gateway currently parses `model` and `stream` after reading the full request body.

With disk replay, this should be revisited:

- Parse only top-level `model` and `stream`.
- Avoid a full JSON unmarshal into memory for very large bodies.
- Preserve the exact original body for upstream forwarding.

### Runtime Cleanup

- Clean stale files under `/app/data/tmp` on startup.
- Ensure temp replay files are deleted on request completion and on client cancellation.
- Add tests for cleanup paths.

### Docker Desktop First Run

- Document a single-image Docker Desktop Run path.
- Default exposed port should be only `7788`.
- First local run should have no required environment variables.
- Document volume mapping as `lune-data` or a host directory mounted to `/app/data`.
- Present `LUNE_ADMIN_TOKEN` as optional for local use and recommended for remote/reverse-proxy use.
- Move `CPA_API_KEY`, `LUNE_CPA_API_KEY`, and `LUNE_CPA_MANAGEMENT_KEY` to advanced/internal documentation.
- Warn users not to publish port `8317`.
- Warn users not to expose `7788` directly to public or untrusted networks.
- Startup banner should stay accurate when the host maps container port `7788` to a different host port; Docker Desktop users may choose a different host port.

### Pool-Scoped Credentials

The v0.1.3 access-token model should be simplified around Pools.

Accepted product direction:

- Remove Global Token as a user-facing concept.
- Do not expose global credentials in Overview, Settings, Pool detail, API snippets, exports, or docs.
- A Pool is the only external client access unit.
- Each Pool has exactly one access token.
- Pool tokens are created automatically and enabled by default.
- Pool tokens cannot be disabled or deleted through the UI.
- Users may reveal, copy, and regenerate the Pool token.
- Regenerating a token immediately invalidates the previous token.

Rationale:

- Codex CLI and editor integrations are configured around a concrete provider and model.
- A global token can route across multiple Pools, which makes client behavior harder to explain and debug.
- Pool-scoped tokens make setup deterministic: a generated client config always targets the Pool shown on screen.
- Settings should no longer ask users to understand token ownership or global-vs-pool scope.

Backend implications:

- Stop creating a default global token on app startup.
- Remove or deprecate `EnsureDefaultGlobalToken`.
- Remove `POST /admin/api/tokens/global/reveal`.
- Remove default UI/API paths for creating, deleting, enabling, or disabling access tokens.
- Keep token rename support because Pool credentials should still show and edit a human-readable token name.
- Keep `POST /admin/api/tokens/{id}/reveal`.
- Keep `POST /admin/api/tokens/{id}/regenerate`.
- Add an `EnsurePoolToken(pool_id)` style backend path so every Pool always has one token.
- Ensure Pool creation always creates its token in the same transaction or immediately after creation with rollback-safe behavior.
- Existing Pools without tokens should be repaired by migration/startup reconciliation.
- Existing global tokens should be hard-deleted during migration/reconciliation.
- Existing Pools with multiple tokens are treated as a legacy edge case outside the default v0.1.3 product path; v0.1.3 implementation may keep one survivor, but broader multi-token conflict UX is deferred.
- Gateway routing should treat every token as Pool-scoped.
- `GET /v1/models` should return models visible to the authenticated token's Pool, not all models globally.
- Request logs should continue recording token name and Pool ID.

Database implications:

- `access_tokens.pool_id` should become conceptually required.
- Consider adding `UNIQUE(pool_id)` to enforce one token per Pool.
- The physical `enabled` column may remain during v0.1.3 for migration compatibility, but application behavior should treat Pool tokens as enabled.
- The `is_global` computed field should be removed from the frontend contract once no API returns global tokens.

Migration policy:

- Hard-delete `pool_id IS NULL` access tokens.
- For each Pool, keep or create exactly one token.
- If a Pool already has exactly one token, keep it.
- If a Pool has no token, create one.
- If a Pool has multiple tokens, keep the lowest ID token for v0.1.3 and delete extras; richer conflict handling is deferred.
- Create a new token for every Pool with no survivor.
- Invalidate routing cache after reconciliation.

Deferred token-scope follow-up:

- If Lune ever needs multiple tokens per Pool again, design it as a later version with explicit use cases, limits, rotation UX, and audit display.

### Settings Credential UI

Settings should replace `Token Management` with a Pool-centered credentials section.

Target section:

- Section title: `Pool Credentials` or similar.
- One row per Pool.
- Each row shows Pool name, masked token, last used time, and actions.
- Actions:
  - Copy
  - Reveal / Hide
  - Edit name
  - Regenerate
- Remove:
  - `Global Tokens`
  - `Pool Tokens` grouping
  - `Create Token`
  - ownership selector
  - enable / disable
  - delete token

Design notes:

- Keep the section dense and consistent with current Settings tables.
- Do not add a large explanatory card.
- Avoid nested cards.
- Use compact rows with icon actions and short labels.
- If a Pool token is missing because reconciliation failed, show an inline repair/error state rather than an empty token-management workflow.

### Pool Detail Client Setup

Pool detail should become the primary client setup surface.

Replace the current `Env Snippets` button with a Pool-scoped client setup action:

- Preferred labels: `Setup`, `Client Setup`, or `Codex Setup`.
- The action should only use the current Pool token.
- Do not fall back to a global token.
- If the Pool token is missing, show an error such as `当前 Pool 凭证未就绪，请刷新或重新生成。`

Remove the current `QR` action:

- The current `lune://connect` payload has no in-repo consumer.
- QR is not a natural fit for Codex CLI or editor configuration.
- A text/download setup workflow is clearer and verifiable.

The replacement setup dialog should provide practical client configuration:

- `Codex CLI` as the primary tab.
- `VSCode` may appear as a disabled or placeholder tab if it fits the visual design, but it should not ship as a complete setup flow in v0.1.3.
- Do not include Shell or curl setup tabs in v0.1.3.
- Provide copy actions for environment variables and config blocks.
- Provide a browser-side download for `config.toml`.
- Provide the target path hint `~/.codex/config.toml`.
- Warn users to merge with an existing config rather than blindly overwriting it.

Suggested Codex CLI environment variable:

```bash
export LUNE_POOL_<pool-id>_API_KEY="sk-lune-..."
```

Suggested Codex CLI config:

```toml
model = "gpt-5.5"
model_provider = "lune-pool-<pool-id>"

[model_providers.lune-pool-<pool-id>]
name = "Lune / <pool name>"
base_url = "http://127.0.0.1:7788/v1"
env_key = "LUNE_POOL_<pool-id>_API_KEY"
wire_api = "responses"
```

Generation rules:

- `base_url` should use the current browser origin plus `/v1` unless the user has configured an external URL.
- `model` should default to the newest/latest discovered model in the current Pool.
- Newest model selection should prefer known GPT version ordering when possible, for example `gpt-5.5` over `gpt-5.1` over `gpt-5` over `gpt-4.1`.
- If models cannot be compared by known version ordering, fall back to the last discovered Pool model.
- If the Pool contains no discovered models, use a conservative placeholder and make it obvious the user should replace it.
- Provider names and env var names must be stable, shell-safe, and derived from Pool ID rather than label.

Placement decision:

- The Setup action only lives on Pool detail in v0.1.3.
- Settings `Pool Credentials` should remain credential management only.

Implementation decisions:

- Keep `PUT /admin/api/tokens/{id}` for token rename only.
- The rename endpoint must ignore or reject `pool_id`, `enabled`, and `token` mutations.
- Backend owns Pool token repair:
  - run reconciliation on startup,
  - ensure Pool token creation during Pool creation,
  - and defensively ensure a token when serving Pool detail or Pool token APIs.
- Frontend never creates a Pool token directly; it can only surface a missing-token repair/error state if backend reconciliation fails.

### Overview Credential Surface

Overview should no longer show global API/key controls after Global Token is removed.

Required changes:

- Remove `StardustGlobalAccess` or redesign it so it does not imply global credentials.
- Remove global token copy, global snippets, and global QR flows.
- Keep Overview focused on status, metrics, alerts, and navigation into Pools.

Potential replacement:

- A subtle Pool navigation shortcut may be added later, but v0.1.3 should avoid recreating global access semantics under another name.

## Implemented Fixes To Mention In Release Notes

These fixes were added during v0.1.3 implementation and should be reflected in the version update summary.

### Account Card Refresh

- The account card action formerly labeled `刷新模型` is now labeled `刷新`.
- The refresh action is a first-level card action beside `查看详情`, not hidden inside the more menu.
- Manual refresh still discovers the account model list.
- For Codex CPA accounts, manual refresh also triggers an immediate Codex quota fetch.
- Manual quota refresh bypasses the background quota fetch throttle; the background loop still uses `codex_quota_fetch_interval`.
- If model refresh succeeds but quota refresh fails, the UI reports the quota failure without discarding the refreshed model result.

### Codex CPA Expiry Semantics

- Codex CPA `cpa_expired_at` represents the volatile CPA credential/token expiry, not the user's ChatGPT subscription expiry.
- Codex CPA credential expiry must not be shown as "account expires in N days" on cards, account headers, Overview alerts, or system notifications.
- Codex CPA should not emit `account_expiring` alerts based on `cpa_expired_at`.
- Codex CPA subscription expiry must be stored separately as `cpa_subscription_expires_at`.
- Lune should fetch Codex subscription expiry through CPA management `api-call` against ChatGPT account-check data, then parse known subscription/entitlement expiry fields.
- Subscription parsing must first scope the response to the matching ChatGPT account ID or account-keyed map entry; it must not fall back to unrelated root-level data when the target account cannot be found.
- If multiple subscription-like expiry fields exist in the same account payload, parsing must use a deterministic field priority instead of map iteration order, so the displayed remaining days do not jump between competing timestamps.
- Codex CPA cards, account headers, Overview alerts, and system notifications should use `cpa_subscription_expires_at` when it is available.
- If the ChatGPT endpoint does not return a recognizable subscription expiry, Lune should show no subscription expiry rather than falling back to credential expiry.
- Codex quota fetch should not skip a Codex CPA account solely because `cpa_expired_at` is in the past.
- Account detail Debug may show both `ChatGPT Subscription Expires` and `CPA Credential Expires`, with distinct labels.
- This avoids misleading day-count jumps such as an account appearing to expire in 7 days and later in 10 days after credential refresh behavior changes.

### CPA Credential Needs-Login State

- `cpa_expired_at` should be treated as the current access-token expiry/debug signal, not as a user-facing "must re-login" deadline.
- Lune should persist a separate CPA credential state:
  - `cpa_credential_status`: `unknown`, `ok`, or `needs_login`.
  - `cpa_credential_reason`: such as `disabled`, `file_missing`, `file_corrupt`, `auth_index_missing`, `auth_failed`, or `refresh_failed`.
  - `cpa_credential_last_error` and `cpa_credential_checked_at` for diagnostics.
- CPA credential status should become `needs_login` when:
  - the auth file is missing or corrupt;
  - the auth file is marked `disabled`;
  - CPA management cannot find the auth index for the account;
  - CPA provider/model checks, quota/subscription `api-call`, or live gateway requests return 401/403 or explicit token/refresh auth failures.
- Successful CPA model checks, quota/subscription checks, or live gateway responses may mark the credential state back to `ok`.
- Account cards and account details should display a red `需要重新登录` badge only when `cpa_credential_status = needs_login`.
- Debug should show CPA credential status/reason/error/check time, and label `cpa_expired_at` as token expiry rather than subscription expiry.
- Overview and notification settings should use a dedicated `cpa_credential_error` event instead of overloading `account_expiring`.
- The default notification text should clearly say the CPA login state is invalid and the user needs to log in again.

## Deferred Security Follow-Up

The current admin auth behavior trusts loopback and private-network sources to make Docker Desktop local access smooth. This is acceptable for the v0.1.3 local-first path only when the published port is bound to localhost or otherwise protected.

A later release should tighten this model for remote deployments:

- Provide an explicit setting for local-trust behavior.
- Require admin authentication for non-loopback sources by default, or make private-network trust opt-in.
- Make Docker/Compose defaults continue to work for local users.
- Document safe reverse-proxy and VPN deployment patterns.
- Add tests that cover localhost, Docker bridge, private LAN, and public remote address behavior.

## Non-Goals For Discussion Phase

- Do not rewrite CPA.
- Do not remove existing CPA account features.
- Do not commit to a breaking migration before deciding the release shape.
- Do not require users to run Compose for the default full experience.
